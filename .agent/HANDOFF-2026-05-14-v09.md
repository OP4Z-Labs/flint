## Completed in this run

### Phases shipped

1. **Manifest foundation** — `flint.manifest.json` writes on every
   `init`, `create-app`, and `add` invocation. Records every file with
   `templateVersion`, `templateSource`, `sha256`, `modified`, `ejected`,
   plus a `vars` map (so `flint upgrade` can re-render against the
   original scaffold args) and a `history[]` of every prior Flint
   invocation. Schema version 1; locked via unit tests.

2. **`flint upgrade --check / --diff / --apply / --dry-run`** — the
   core v0.9 feature. Walks the manifest, classifies every entry into
   one of four states (`unmodified` / `modified` / `ejected` / `missing`),
   and offers:
   - `--check`: drift table; exit 1 if any drift, 0 if clean.
   - `--diff`: unified diffs for modified files.
   - `--apply`: interactive 3-way merge per drifted file with options
     `keep / take-new / merge (via $EDITOR) / eject`. Unmodified files
     auto-update silently to the new template. Missing files prompt for
     `restore / remove / skip`.
   - `--dry-run`: like `--apply` but writes nothing.
   - **Backfill mode**: when a project has no manifest (scaffolded by
     Flint ≤ 0.5), upgrade auto-builds one from the current file state,
     using a sentinel `sha256` (all zeros) so every entry classifies as
     `modified` until the user explicitly reconciles them.

3. **"Generated by flint@X.Y.Z" header comments** — every scaffolded
   file that supports comments now carries the version header at the
   top. The `flintVersion` template variable is injected into the
   render dict; on upgrade, the CURRENT version (not the originating
   one) substitutes so reapplied templates carry the upgrading version's
   tag. Files converted to `.tmpl` for this purpose:
   - `_skeleton/eslint.config.js → eslint.config.js.tmpl`
   - `{variant}/public/_headers → _headers.tmpl` (all 3 variants)
   - `{variant}/.github/workflows/ci.yml → ci.yml.tmpl` (all 3 variants)
   - Existing `.tmpl` files (wrangler.toml, vite.config) had their
     headers rewritten to the canonical "Generated by flint@X.Y.Z — run
     `flint upgrade --check` for drift" form.

4. **`flint add pwa | auth | rate-limit`** — three new feature scaffolds:
   - `add pwa`: installs `vite-plugin-pwa` + `workbox-window` if missing
     (auto-detects npm/pnpm/bun/yarn from lockfiles); patches `vite.config.ts`
     to add the `VitePWA(...)` plugin block; idempotent (detects existing
     `vite-plugin-pwa` reference and bails out).
   - `add auth`: drops `functions/_shared/auth.ts` (from the pages-fullstack
     template) and merges `ADMIN_PASSWORD` + `COOKIE_SECRET` stubs into
     `.dev.vars.example`. Preserves existing entries.
   - `add rate-limit`: drops `functions/_shared/ratelimit.ts`. Requires
     a KV binding; auto-prompts to create `RATELIMIT_KV` via `add kv`
     if none exists.

5. **`flint auth purge`** — the safety command from the 2026-05-14 manual
   smoke. Wipes `~/.config/flint/credentials` + `.dev.vars` (only if it
   has `CLOUDFLARE_API_TOKEN`); clears OS keychain entries when keychain
   backing is active; with `--include-archive` also nukes the rotated
   credentials archive. Always prints the dashboard URL with a prominent
   "revoke the token manually" reminder, because the Cloudflare API
   cannot self-revoke tokens.

6. **`flint auth init --keychain`** — opt-in OS keychain backing for
   credentials. Loads `keytar` lazily at runtime (no native dep in the
   default install path); falls back gracefully to `.dev.vars` if keytar
   isn't available (headless Linux without libsecret, etc.). Persists
   the storage choice in `~/.config/flint/storage-mode`. `flint auth status`
   now reports which mode is active.

7. **`flint create-app --template git+<url>`** — real custom-template
   support. Parses `git+https://host/u/r[#ref][/subdir]` and the
   `@ref` npm-style variant; shallow-clones into a scratch dir; copies
   contents into the target (sans `.git/`); records every file into the
   manifest with `templateSource = git+<url>/<rel>`. Custom-template
   files are correctly excluded from `flint upgrade` template re-rendering
   (no bundled source to render against).

8. **Telemetry first ship** — opt-in, off by default, **local log only**
   for v0.9. First-run consent prompt; preference at
   `~/.config/flint/telemetry.json`; `flint config --telemetry on|off`
   to toggle. Events written to `~/.config/flint/telemetry.log`. Event
   shape is locked in unit tests — see "Notes for the next agent" for
   the public-ish contract.

9. **Integration tests** — +30 across the new surfaces.

10. **README + roadmap** — updated to reflect v0.9 surface; manifest
    schema and telemetry event shape documented as public-ish contracts
    that v1.0 must preserve.

### Design decisions taken

1. **Manifest is separate from `flint.config.json`.** The brief's
   wording made the distinction explicit: manifest is forensic state
   (Flint-managed, never hand-edited), config is preferences (user-
   editable). Mixing them would mean we can't safely overwrite the
   manifest on every Flint run.
   *Alternative considered:* one merged file with separate sections.
   *Rejected because:* a single file means any read needs to preserve
   user-section round-trips on write. Two files = two distinct
   contracts with simpler semantics each.

2. **Backfilled manifest uses a sentinel sha (all zeros), not the
   current file's sha.** The brief said "flag everything as modified"
   — the simplest implementation would record the current sha + set
   `modified: true`, but classifyFile recomputes sha from disk on every
   call, so the cached flag is overruled. Using a sentinel sha makes
   the classifier truthfully report `modified` until an explicit
   `upgrade --apply` reconciles each file.
   *Why this matters:* the conservative default is "never auto-overwrite
   anything backfilled" — silent updates on backfill would be a footgun.

3. **`upgrade --apply` "keep" choice resets the baseline sha.** When the
   user chooses to keep their version, we record THEIR sha as the new
   baseline. Subsequent `upgrade --check` runs won't keep flagging the
   same drift. The templateVersion is NOT advanced (user owns the
   content), but at least the noise is removed.
   *Alternative considered:* leave the baseline alone. *Rejected because:*
   the user would see the same drift on every upgrade — eventually they'd
   eject just to make it stop. Eject is the "I want full ownership"
   choice; keep is "I'm fine with my version, stop bothering me."

4. **`eject` preserves the user's current sha AND sets `ejected: true`.**
   Ejected entries are tracked but never modified by upgrade. We still
   record their sha so future commands like `add` or another `eject`
   re-run have an accurate snapshot.

5. **Editor merge writes a conflict-marker file.** `$EDITOR` opens a
   file with `<<<<<<< Yours / ======= / >>>>>>> Flint` markers, the user
   resolves, saves. The standard `git mergetool` pattern. No structured
   3-way diff parser — keep it simple, the user owns the resolution.
   *Why:* a structured merge would either need a heavyweight dep (`diff3`)
   or hand-rolled hunk reconciliation that produces worse output than a
   competent text editor.

6. **Keychain backing is lazy-loaded via dynamic-import.** Hard-pinning
   `keytar` means a fresh `npm install flint` drags in native bindings
   that fail on minimal CI containers. `tryLoadKeytar()` returns null on
   any of (a) keytar not installed, (b) keytar loaded but probe failed,
   (c) `FLINT_KEYCHAIN_DISABLED=1`. The caller falls back to file-based
   storage with a clear "keychain not available" warning.
   *Cost:* users who want keychain backing have to `npm install -g keytar`
   themselves. Acceptable — most have it for 1Password CLI / vault.
   *Test hook:* `FLINT_KEYCHAIN_FAKE=1` activates an in-memory Map-backed
   shim so the keychain test suite is portable across CI environments
   without real libsecret / Keychain access.

7. **Telemetry event shape is locked via tests, not just documented.**
   The brief flagged this as a public-ish API. The test
   `tests/util/telemetry.test.ts` explicitly asserts on field
   presence (`event`, `flintVersion`, `os`, `node`, `ts`) AND field
   absence (`path`, `cwd`, `user`, `email`, `hostname`, `token`,
   `accountId`). Future schema additions add fields — they never rename
   or remove existing ones.
   *Rationale:* v1.0 ships the same shape to a remote endpoint. Users
   writing scripts against the local log in v0.9 must not need rewrites
   when the remote endpoint lights up.

8. **First-run consent prompt is silent-default-off in non-TTY contexts.**
   When stdin isn't a TTY (CI, piped, `--yes` workflows), we persist
   `enabled: false` without prompting. Subsequent runs find the saved
   pref and don't bother the user. The pref is also persisted on
   `FLINT_TELEMETRY_NO_PROMPT=1` for explicit script overrides.

9. **`flint create-app --template` clones via `git`, not a node tarball.**
   Considered `node-tar` + the GitHub tarball endpoint for a stdlib
   approach. *Rejected because:* (a) requires guessing the host's tarball
   URL convention (`/archive/refs/heads/main.tar.gz` on GitHub but
   different on GitLab/Codeberg/SourceForge), (b) breaks for private
   repos that the user has SSH access to, (c) git is universally
   installed on dev machines already. `git clone --depth 1` is one
   spawnSync and handles every protocol git supports natively.

10. **Header comments converted some plain template files to .tmpl.**
    `_headers`, `ci.yml`, `eslint.config.js` got renamed. *Considered:*
    leave them as plain copy-byte files and add the header dynamically
    in the writer. *Rejected because:* the template-walker already has
    one substitution path (`.tmpl` files), and adding a second
    "files-that-need-a-header-prepended" path doubles the surface for no
    real gain. The `.tmpl` extension is also a useful signal to anyone
    eyeballing the templates dir.

11. **`add pwa` detects PWA already wired and bails out.** The
    pages-fullstack and static-spa variants ALREADY ship vite.config.ts
    with VitePWA enabled. `add pwa` on those projects is a no-op — we
    print a friendly "already wired" message rather than churning the
    config. The patch path is meaningful only for projects that opted
    out (e.g. pages-functions variant) or generated their own vite.config.

12. **`add rate-limit` auto-prompts to create a KV binding.** The
    rate-limit pattern needs a KV namespace. We could refuse to install
    without one, but that's a worse UX than asking. In `--yes` mode
    (non-interactive), we refuse with a "run `flint add kv RATELIMIT_KV`
    first" message instead of silently doing it — flint should never
    quietly add Cloudflare resources in autonomous mode.

1. **Telemetry first-prompt copy.** Current text:
   `"Help improve Flint by sharing anonymous usage stats? (no project info, no token info)"`
   *Recommendation:* keep as-is; it's the wording the brief implied.
   *Why:* explicit reassurance about what's NOT collected pre-empts the
   "should I trust this CLI?" reaction. The longer wording costs a line
   but earns user goodwill.

2. **Default for `flint upgrade` without a mode flag.** Currently prints
   an info banner and defers to `--check`.
   *Recommendation:* keep this. Auto-running `--check` on bare
   `flint upgrade` is the least-surprising behaviour — users come in
   expecting "tell me what's drifted" before any state-changing op.

3. **Should `add pwa` warn about an existing service-worker file
   (`public/sw.js`) that wasn't generated by Flint?** Currently it
   doesn't — `vite-plugin-pwa` generates `sw.js` at build time anyway,
   so a stale hand-rolled one will be overwritten.
   *Recommendation:* don't add a warning. The runtime behaviour is
   correct; warning before the fact is noise.

4. **`flint auth purge --include-archive` default.** Currently off.
   *Recommendation:* keep it off. The archive is the recovery path
   for "I rotated and the new token has a problem" — the user explicitly

## Notes for the next agent (v1.0 work)

### Manifest schema details (load-bearing for v1.0 rescaffold)

The Blaze/Chorus/Portfolio rescaffold will read `flint.manifest.json` to
detect "this project was upgraded N times, last template version was X."

Schema fields the rescaffold work depends on (DO NOT rename without ADR):
- `flintVersion` — the version that LAST WROTE the manifest. For a
  long-lived project this advances on every `init`/`add`/`upgrade`.
- `variant` — sticky; set on first scaffold, never rewritten by `add`.
- `vars` — sticky on first scaffold; later `add` invocations don't
  overwrite existing keys (so `add pwa` doesn't accidentally change the
  appName recorded at init time).
- `history[]` — append-only; never pruned. v1.0 may want to compress
  this for very old projects (10+ years of history) but the field
  must remain present.
- `files[].templateSource` — the lookup key for re-rendering. Format
  is `<variant-or-_skeleton>/<rel-path-with-.tmpl-suffix-IF-template>`.
  For git+url templates, it's `git+<url>/<rel>`. The rescaffold work
  needs to handle both shapes.
- `files[].ejected` — once true, never auto-clear. v1.0 might add
  `flint upgrade --unjeject <path>` as an explicit user-driven unset.

### Telemetry event shape (public-ish API once endpoint lands)

The event payload that v0.9 writes to local log is the SAME shape
v1.0 will POST to the remote endpoint. Locked fields:

| Field | Type | When present | Notes |
| --- | --- | --- | --- |
| `event` | string | always | command name, no args |
| `variant` | string | init / create-app | the template variant chosen |
| `errorType` | string | error paths only | error name/code, NEVER message |
| `context` | object | rare | small ad-hoc dict for event-specific data |
| `flintVersion` | string | always | `readPackageVersion()` |
| `os` | string | always | `process.platform` value |
| `node` | string | always | major only, e.g. `"20"` |
| `ts` | string | always | ISO 8601 with milliseconds |

**Explicitly forbidden fields** (asserted in the test, locked PUBLIC contract):
- `path` / `cwd` / project paths
- `user` / `email` / `hostname` / machine identifiers
- `token` / `accountId` / credential material
- `message` — error messages may contain PII

Adding new fields is fine. Renaming or removing existing fields is a
MAJOR-version change.

### Keychain fallback behavior

| Platform | Default keytar backend | Failure modes |
| --- | --- | --- |
| macOS | system Keychain | Locked Keychain → keytar errors → Flint falls back to .dev.vars |
| Windows | Credential Manager | Should work out of the box |
| Linux | libsecret + gnome-keyring | No libsecret installed → no keychain available → fall back |
| Linux (headless CI) | none typically | Always falls back; the `--keychain` flag is essentially a no-op |

The `FLINT_KEYCHAIN_DISABLED=1` env var forces the fallback path
(useful for CI runs that have a desktop session but shouldn't use it).
`FLINT_KEYCHAIN_FAKE=1` activates an in-memory shim — TEST USE ONLY.

### Gotchas encountered

1. **ESM dynamic-import + node:fs interaction.** Wrote `require('node:fs')`
   inside the upgrade backfill helper to fetch readdirSync/statSync from
   within a nested function — that's blocked by
   `@typescript-eslint/no-require-imports` AND by the project's `"type":
   "module"`. Lifted the imports to the top of the file. Same lesson
   applies in the tests — `tests/integration/*` had stray `require`
   calls that lint flagged. Always use top-of-file imports.

2. **`@inquirer/prompts` `select<T>` with typed unions.** The select call
   in `upgrade --apply` needs explicit string union types. Using
   `select<'keep'|'take-new'|'merge'|'eject'>` works; an untyped call
   returns `unknown` and trips strict-mode checks.

3. **`mkdtempSync` returns symlink-resolved paths on macOS.** Tests that
   write a "credentials" file inside the temp home then read it back via
   `fsExistsSync` worked on Linux + Mac in v0.9, but I noticed the
   sample-creds test sometimes saw `/var/folders/...` instead of
   `/private/var/folders/...`. The HARNESS is platform-agnostic; just
   never assume an exact path string when probing.

4. **Headers comment in `.tmpl` files counts as template content.** The
   `# Generated by flint@{{flintVersion}}` line goes through `renderString`
   like everything else — that means a missing `flintVersion` var
   throws. The render-test fixture needed `flintVersion: '0.9.0'` added
   to its canonical-vars dict; tests that don't use the test fixture
   would fail with `"Template variable \"flintVersion\" not provided."`
   if they forgot.

5. **`process.stdin.isTTY` in spawn'd child processes.** Spawned with
   `stdio: ['pipe', ...]`, stdin is non-TTY. The telemetry consent
   prompt correctly skips the prompt in that case and persists
   `enabled: false`. This is a feature, not a bug — integration tests
   never accidentally toggle telemetry on.

### Conventions established

1. **Manifest tracker call pattern**: construct → record per file → flush
   once at end. NEVER persist between records (loses history-entry semantic).
2. **`recordFile` preserves `ejected: true` across re-records.** Ejected
   files stay ejected forever unless explicitly cleared. New writes update
   the cached sha but leave the flag alone.
3. **Header comments**: `# Generated by flint@{{flintVersion}} — run \`flint upgrade --check\` for drift.`
   Use `{{flintVersion}}` template var, never hard-code a version.
4. **Telemetry consent path is non-throwing.** `emitEvent` swallows every
   error — telemetry must never break a user command. The unit test
   "never throws even when filesystem is unwritable" locks this.
5. **`add` subcommands record their writes into the manifest with a
   `command: "add pwa"` (etc.) history entry.** This is the
   distinguishing signal that lets `upgrade --check` group changes by
   provenance.

### Partial-state cautions

1. **`upgrade --apply` writes the manifest at the END of the walk.** If
   the user Ctrl-C's mid-walk, the manifest is not updated even though
   some files may have been written. This is intentional — partial
   writes are recoverable (re-run upgrade and the same files will be
   flagged again), but a partially-updated manifest would be
   inconsistent forever.

2. **`--template git+<url>` strips `.git/` AND `flint.manifest.json` from
   the cloned tree.** The new project starts with a fresh manifest. If
   a future feature wants to RESPECT an upstream manifest (e.g. for
   "templates that are themselves Flint projects"), the strip needs to
   become conditional.

3. **`auth purge` does not revoke the token in Cloudflare.** Always.
   Always. The Cloudflare API does not let a token revoke itself.
   The dashboard reminder is THE feature, not just a nicety. Future
   work could add an `--also-rotate` flag that walks the user through
   creating a replacement before purging, but the current scope is
   strictly "clean up local state."
