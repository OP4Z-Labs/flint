# Flint v0.5 — Create-app + Deploy Handoff (2026-05-14)

Ships `flint create-app` (full new-project bootstrapping for all three
template variants), `flint deploy` (wrapped deploy with pre-flight +
asset-budget guard + health-ping + rollback UX), the new `static-spa`
variant (Portfolio-parity), and the shared Vite + React + TS skeleton
that backs every variant. All scope items in the v0.5 brief are done.

### Design decisions taken

1. **Skeleton + variant composition over per-variant duplication.**
   *Alternative considered:* duplicate every common file (tsconfig,
   eslint config, vitest config, package.json, src skeleton) under
   each variant tree. Rejected because the v0.5 brief explicitly
   warned against this ("don't fabricate; copy real file shapes")
   and because a future change to the React/Vite version base would
   need to be made in three places.
   *Chosen:* `templates/_skeleton/` ships the common files;
   `templates/<variant>/` ships the variant-specific overlay
   (wrangler.toml, vite.config, functions/_shared, etc.). The
   `create-app` command lays skeleton down first, then variant
   on top — variant wins on conflicts. `flint init` (existing-repo
   overlay command) is **not** changed; it continues using the
   per-variant trees only, since it's overlaying onto an EXISTING
   Vite scaffold and shouldn't touch tsconfigs / package.json.

2. **`flint.config.json` is introduced for asset-budget only.**
   The plan §10 open question 8 said "defer until v0.9." We've
   introduced it earlier, in minimal form, because the deploy
   surface is the first place users will genuinely want to tune
   thresholds. **Only `assetBudget.maxBundleMB` and
   `assetBudget.maxChunkKB` are read.** Everything else is still
   driven by `wrangler.toml` / `package.json` / `.dev.vars`. The
   broader config story stays deferred. Defaults (5 MB / 500 KB)
   are the public API once shipped — see "Notes for the next agent"
   for the rationale.

3. **`--rollback` does not auto-fire on health-ping failure.** The
   v0.5 brief asked specifically about this. Chosen behavior: print
   the rollback command for one-paste recovery, but let the user
   trigger it. *Alternative considered:* auto-rollback when GET /
   returns non-2xx within N seconds. Rejected because:
   - A real deployment can take 30-60s to warm up on Pages — a
     transient ping failure during cold-start would trigger a
     reflexive rollback.
   - Users with health endpoints behind auth (admin-only) would
     ALWAYS fail the ping; auto-rollback would never let them deploy.
   - Rolling back deletes work. Asking is cheap; making the wrong
     call automatically is expensive.

4. **Wrangler runs through the existing wrangler-runner adapter.**
   The deploy command shells out via `runWrangler(...)` from
   `src/cloudflare/wrangler-runner.ts` — same plumbing as v0.2's
   configure command. The runner already handles
   `WRANGLER_BINARY` env override (testing hook), node-modules-bin
   resolution, and CLOUDFLARE_API_TOKEN env injection.

5. **`--skip-checks` does NOT bypass build.** Build is the only step
   that produces `dist/`, and without `dist/` there's nothing for
   `wrangler pages deploy` to push. The flag explicitly applies to
   lint / typecheck / vitest only.

6. **Pre-flight FAILURE paths are the integration-test focus.**
   The deploy hot-path (lint → build → fetch CF → wrangler push)
   requires a real network + real wrangler. We integration-test
   only the early-exit paths (missing wrangler.toml, missing
   credentials, rollback UX) and unit-test the exported parsers
   (`parseDeployStdout` / `parseDeploymentList`). The wrangler
   stand-in uses the same POSIX fake-bin pattern as v0.2's
   wrangler-runner unit tests, via `WRANGLER_BINARY` env override.

7. **Package manager detection via `npm_config_user_agent`.** This
   is the canonical signal used by create-vite, create-next-app,
   and shadcn-ui. Other approaches considered + rejected:
   - Sniffing lockfiles: doesn't apply to create-app (target dir
     doesn't exist yet).
   - `which pnpm` / `which bun`: gives "is it installed", not
     "which one invoked us." Falls back to noisy heuristic ordering.
   - `process.argv0`: not reliable across PMs.
   The UA-string parser also recognizes yarn but returns `null`
   for it — yarn is not a v0.5 target, so we fall back to npm.

8. **`git init` writes the default branch to `main`.** Uses `git
   symbolic-ref HEAD refs/heads/main` (best-effort; ignores failures
   for `init.defaultBranch=main` configs that already do this).

9. **No new runtime dependencies.** All v0.5 functionality uses
   Node stdlib (`node:zlib`, `node:fs`, `node:child_process`,
   `node:crypto`) + the existing `@inquirer/prompts` + `commander`
   + `smol-toml` deps from v0.2. **`gzipSync` from `node:zlib`**
   is the load-bearing choice — measures real per-chunk wire
   size, available since Node 14.

## Open questions for the user

1. **Asset-budget threshold defaults.** Chose 5 MB total / 500 KB
   gzipped-per-chunk. Once shipped these become public API (changing
   them is a minor-version bump that triggers warnings in user
   builds that previously passed).
   *Recommendation:* keep 5 / 500. 500 KB gzipped is the
   industry-standard "huge chunk, look at this" line (matches
   Lighthouse's perf warnings and Cloudflare Pages' soft signal).
   5 MB is a generous total ceiling — most real apps are well under
   that even with images committed to `public/`.
   *Why:* tighter defaults would surface false-positive warnings on
   day-1 deploys of perfectly normal apps.

2. **Should `create-app` warn when running on an old Node version?**
   The skeleton's `package.json` declares no `engines` field. Node
   20+ is the implicit floor (matches all dev deps).
   *Recommendation:* don't add a runtime check; the dev deps'
   `engines` fields will surface a warning at `npm install` time.
   Belt-and-suspenders here adds code without adding signal.
   *Why:* the user said "make the reasonable call and continue."

3. **`flint.config.json` location / format.** Chose JSON at project
   root (alongside `wrangler.toml`). Considered TOML for consistency
   with wrangler.toml; rejected because every other Flint-managed
   surface (package.json, .dev.vars, _routes.json, etc.) is JSON,
   not TOML, and JSON has native `JSON.parse`.
   *Recommendation:* keep JSON. v0.9's broader config can introduce
   TOML support if a real need surfaces.

4. **Should `--provision` be the default?** Currently off. Users who
   want to provision immediately after scaffold pass `--provision`.
   *Recommendation:* keep it off. Most users want to inspect the
   scaffolded files before committing to Cloudflare API calls.
   *Why:* a default-on `--provision` would mean every `create-app`
   needs a valid stored CF token + Account ID — bad UX for users
   who're just trying out the variant.

5. **Pages-functions variant in create-app: same vite.config.ts as
   pages-fullstack?** Currently yes — they share the variant overlay
   pattern but pages-functions uses the pages-fullstack vite.config
   (which has the PWA + /api proxy). The static-spa variant gets
   its own (PWA-on, no /api proxy).
   *Recommendation:* leave it as-is. Pages-functions apps DO need
   the /api proxy if Vite is hit directly; the PWA-on is harmless.
   *Why:* one less template file to maintain; the vite.config is
   identical between the two function-bearing variants today.

## Notes for the next agent

### Asset-budget threshold rationale (public API once shipped)

The 5 MB / 500 KB defaults are PUBLIC API — changing them is a
behaviour-change that retroactively warns on user builds that
previously passed.

**500 KB gzipped per chunk** comes from:
- Lighthouse "JavaScript execution time" budget threshold at the
  green tier (post-gzip).
- Cloudflare Pages' Worker size budget (25 MB total raw script
  size) divided across a reasonable code-split set of ~50 chunks.
- Industry custom (webpack-bundle-analyzer, vite-plugin-pwa warn
  thresholds, Next.js First Load JS).

**5 MB total dist/** comes from:
- Median real-world dist/ for a Vite + React + TS + Tailwind +
  PWA app sits around 1-2 MB (most of it images in `public/`).
- 5 MB is enough headroom that legitimate "I'm hosting a few
  large images" cases don't fire, but tight enough that a
  bundle-size regression (e.g. accidentally pulling in moment.js
  + lodash + chart.js) WILL fire.

If we ever tighten these, the migration path is: keep the warn
behavior (no `--strict-budget` change) but provide a `flint upgrade
--apply` hook that auto-bumps the user's `flint.config.json`
to the previous defaults.

### Patterns established that v0.9 work should follow

When `flint upgrade` lands, follow these conventions:

1. **Read templates the same way create-app does.** The skeleton +
   variant composition + `.tmpl`-suffix-stripping logic is shared
   between `init` and `create-app` today. Don't roll a third
   template-walker; extract a helper if needed.
2. **Generated-file header comments.** Every templated file should
   carry a `# Generated by Flint. Edit freely — Flint will warn
   before overwriting on `flint upgrade`` line so the upgrade
   diff has an obvious provenance signal.
3. **Drift detection via line-based comparison.** The wrangler-toml
   patcher uses surgical line-range edits to preserve comments;
   `flint upgrade` should use the same primitive for any config
   files it touches.

### Gotchas encountered

1. **`@inquirer/prompts` and `select<T>` typing.** When the inquirer
   select expects `<string>`, the choices array's `value` must be
   `string`. Don't pass a typed union directly; cast at the call
   site if the value type is narrower than string.
2. **`require()` in tests.** ESLint flags `@typescript-eslint/no-require-imports`
   if you sneak a `require('node:fs')` into a `.spec.ts` file —
   even though the rest of the codebase is ESM. Use top-of-file
   `import { mkdirSync as mkdirSyncFs }` aliases instead.
3. **Logger output streams.** `log.warn()` and `log.err()` write to
   stderr (via `console.warn` / `console.error`), not stdout.
   Integration-test assertions on warnings must check the
   combined stdout+stderr stream.
4. **Vitest `--reporter=verbose` shows 172 tests but `npm test`
   shows 195** — turned out to be a stale build artifact. Always
   `npm run build` before running the integration tests if you've
   changed source.

### Conventions established

1. **Spec naming.** `create-app.spec.ts` for the surface in general;
   `static-spa.spec.ts` for the variant-contract guarantee; not
   `pages-fullstack.spec.ts` / `pages-functions.spec.ts` (those
   are covered by the cross-variant create-app spec). Add a
   variant-specific spec ONLY when there's a public-contract
   guarantee that diverges from the broad happy path.
2. **POSIX fake-bin pattern.** When testing a command that shells
   out to `wrangler`, write a `#!/bin/sh` script with the canned
   output, `chmod +x`, point `WRANGLER_BINARY` at it. POSIX-only
   tests are documented as such.
3. **`--no-install --no-git --yes` is the integration-test trio.**
   Every create-app test passes all three; this keeps tests fast
   (~100ms per spawn instead of ~5s) and avoids real-network
   pulls or stranded `.git` directories.

### Partial-state cautions

1. **`flint create-app` rejects existing non-empty target dirs.**
   The "occupied directory" guard checks for "meaningful" entries
   (everything except `.git`, `.DS_Store`, `.idea`, `.vscode`).
   If a future scaffold ever needs to merge into an existing
   `.github/workflows` directory, this guard needs adjustment.
2. **The `--template <git+url>` stub is on a one-way ratchet.** The
   warning message refers users to v0.9. When v0.9 lands, the
   warning must be removed AND the actual git-clone logic added
   in the same release.
3. **`flint.config.json` shape is a public contract.** The shape
   today is `{ "assetBudget": { "maxBundleMB": N, "maxChunkKB": N } }`.
   v0.9's broader config additions must keep this nested structure
   (don't flatten to top-level keys like `maxBundleMB` — that
   would clash with a future `appName` / `defaultVariant` etc.).
